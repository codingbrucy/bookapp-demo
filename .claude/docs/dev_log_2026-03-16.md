# Dev Log — 2026-03-16

## Feature: Intake Workflow (End-to-End)

### What Was Built
A complete intake management workflow for the facility admin panel — from the pending requests sidebar on the dashboard through a dedicated intake queue, full patient detail view, and bed assignment modal. Accepting an intake updates the dashboard stats in real time via shared React context.

---

## New Flow

```
Dashboard sidebar → [Review] or [Accept]
                          ↓
               /intakes/[id]  (full patient detail)
                          ↓
               [Accept & Assign Bed] → BedAssignmentModal
                          ↓
               Confirm → patient admitted → navigate to dashboard
               (stats: pending −1, available beds −1, active patients +1)

Dashboard stat card (Pending Requests) → /intakes
Dashboard "View X more" button → /intakes
Nav "Intakes" link → /intakes
```

## Data Architecture

### State (`FacilityContext`)
```typescript
type FacilityContextValue = {
  intakes: IntakeRequest[];
  rooms: Room[];
  acceptIntake: (intakeId, roomId, bedIndex) => void;
  declineIntake: (intakeId) => void;
};
```

### `acceptIntake` side effects
1. Sets `intake.status = "accepted"`
2. Marks selected bed as `occupied: true`, assigns `patientName`
3. Dashboard stats re-derive automatically (no extra logic needed)

### Room data shape change
Rooms moved from `{ totalBeds, occupiedBeds, patients[] }` to `{ beds: Bed[] }` where each bed tracks its own occupancy and patient name. This enables individual bed assignment.

## Design Decisions

**Why React Context (not URL state or localStorage)?**
Context is the standard React pattern for sharing state across a component tree. It avoids serialization overhead of localStorage and keeps the state in-memory for the demo. For production, this would be swapped for API calls.

**Why extract the Navbar to layout?**
The Navbar is now used across 3+ routes. Project convention says extract to `components/` when reused. Putting it in layout is even cleaner — it renders once and is never duplicated. It needs context access (active patient count), so the layout wraps it inside `FacilityDataProvider`.

**Why both Accept and Review go to the same `/intakes/[id]` route?**
The Accept flow requires a bed assignment step, so it can't happen inline on the dashboard without a multi-step modal. Routing both buttons to the detail page simplifies the dashboard and keeps all admit logic in one place.

**Why no toast/success banner after admitting?**
The dashboard stats updating in real time IS the visual confirmation. Adding a separate toast would be redundant noise for the demo.
