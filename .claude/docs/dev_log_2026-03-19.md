## New/Modified Components

### `components/SunMoonToggle.tsx` (rewritten)
Two separate SVG layers inside a wrapper that rotates 180° on toggle:

| Layer | Light Mode | Dark Mode |
|---|---|---|
| **Sun** | Visible — orange circle + 8 rays with `spin-slow` (20s rotation) | Faded out (`opacity-0`) |
| **Moon** | Faded out (`opacity-0`) | Visible — crescent path + 3 small stars, counter-rotated 180° so it appears upright |

The crescent moon is a proper SVG `<path>` using two arcs (`M22 6 A14 14 0 1 1 22 34 A10 10 0 1 0 22 6Z`), not a circle-with-mask hack. Three small `<circle>` stars at different sizes/opacities add detail.

**Rotation**: The outer `<div>` wrapper has `transition-transform duration-700 ease-in-out` and toggles between `rotate(0deg)` (light) and `rotate(180deg)` (dark). The moon SVG has `transform: rotate(180deg)` baked in so it counter-rotates and appears right-side-up after the parent's 180° spin.

## Design Decisions

**Why `#38BDF8` (sky blue) for dark mode primary?**
Pairs well with the purple accent (`#A78BFA`), has strong contrast on dark backgrounds, and feels modern/premium. The original teal appeared muddy on dark surfaces.

**Why `color-mix()` instead of separate opacity variables?**
`color-mix(in srgb, var(--primary) 10%, transparent)` is supported in all modern browsers and lets us derive any opacity variant from a single variable. No need to maintain 10 separate `--primary-10`, `--primary-20`, etc.

**Why the room number hover now uses `var(--accent)` instead of hardcoded amber/purple?**
The accent color already swaps between themes via CSS variables, so the gradient rule works for both light and dark without a separate `.dark` override.

**Why counter-rotate the moon SVG?**
The parent wrapper rotates 180° on toggle. Without counter-rotation, the moon would appear upside-down. The `transform: rotate(180deg)` on the moon SVG cancels out the parent rotation so the crescent always points the right way.
